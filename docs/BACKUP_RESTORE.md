# Backup Restore Testing

Procedure for verifying that daily backups are actually restorable.

---

## Backup structure

Daily backups land at:

```
~/Backups/wilkesliberty/daily/
  YYYY-MM-DD_HHMMSS/
    database/
      drupal_postgres_TIMESTAMP.sql.gz   ← pg_dump, gzip-compressed
    files/
      drupal-files-TIMESTAMP.tar.gz      ← Drupal public/private files
```

Generated by `~/Scripts/backup-onprem.sh`, scheduled via launchd (`com.wilkesliberty.backup`) at 2:00 AM daily.

---

## Running the restore test

```bash
# Test the latest daily backup (most common)
make test-backup-restore

# Test a specific backup directory
scripts/test-backup-restore.sh --backup-dir ~/Backups/wilkesliberty/daily/2026-04-23_020001
```

The script:
1. Finds the newest `drupal_postgres_*.sql.gz` in the backup directory
2. Checks gzip integrity (`gunzip -t`)
3. Spins up a temporary `postgres:16` Docker container
4. Restores the dump via `psql`
5. Verifies: ≥50 tables, all core Drupal tables present, node count > 0, config entries
6. Tears down the container on exit (success or failure)

**Exit codes:** `0` = pass, `1` = fail

**Requirements:** Docker must be running.

## Previously known issue: 20-byte dumps from launchd (FIXED 2026-04-23)

The daily launchd job was producing empty (≈20 byte) `.sql.gz` files on every run since initial deployment. Root cause: launchd's minimal PATH (`/usr/bin:/bin:/usr/sbin:/sbin`) does not include the Docker CLI at `/opt/homebrew/bin/docker` or `/usr/local/bin/docker`.

**This is fixed.** The following were applied on 2026-04-23:
- `backup-onprem.sh` now exports `PATH="/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin"` at the top
- `com.wilkesliberty.backup.plist` has an `EnvironmentVariables` block with the same PATH
- `check_prerequisites()` runs before any I/O — aborts with a Postmark alert if docker or wl_postgres is unavailable
- Dumps go to `.tmp` first; size-check (< 10 KB → failure + Postmark alert); only `mv` to final name on success
- Any failure sends a Postmark email alert to `jmcerda@wilkesliberty.com` with the reason and recent error log tail

If you see 20-byte dumps after this fix, check `~/Backups/wilkesliberty/logs/backup-error.log` — a Postmark alert should also have been sent. See `docs/SECURITY_CHECKLIST.md §6.8` for the full evidence.

## Quarterly restore drill

Schedule a restore test at least quarterly. Steps:

1. `make test-backup-restore` — confirms the automated backup is valid
2. Verify the node count matches the expected site content (not zero)
3. Spot-check config count (should be ≥ 2000 for a populated Drupal install)
4. If any check fails, run `~/Scripts/backup-onprem.sh` manually and re-test

## Backup encryption

Backups are encrypted with AES-256 before Proton Drive sync. The encryption key is in `app_secrets.yml` (SOPS-encrypted) as `BACKUP_ENCRYPTION_KEY`. The restore test operates on the **unencrypted local copy** — it does not test the encrypted Proton Drive copy.

To test the encrypted copy:
```bash
# Decrypt the Proton Drive backup
openssl enc -aes-256-cbc -d -in /path/to/encrypted.tar.gz.enc \
  -pass pass:"$BACKUP_ENCRYPTION_KEY" | tar -xz
# Then run the restore test against the decrypted database dump
scripts/test-backup-restore.sh --backup-dir /tmp/decrypted-backup
```
